Three Things to Keep in Mind

If you only remember three things about how Claude Code works, make it these.

1. The folder you open = Claude's entire world

Claude can only see files inside the folder you opened in VS Code. Switch folders to switch what it sees. If Claude says "I can't find that file," it usually means you need to open the right folder first.

2. One conversation per lesson

Start a fresh chat for each activity. Long, mega-conversations that span multiple lessons get confused about which context matters. Clean slate = cleaner output.

3. Esc is your friend

If Claude starts going in the wrong direction, interrupt it. You don't have to let it finish a bad answer. Press Esc, then redirect — "actually, let's do X instead."

Your VS Code Workspace

A workspace is just "the folder that's open right now." Saving it means you can jump back to it in one click.

Opening a folder

File → Open Folder  (Cmd + O)
Navigate to Google Drive → Shared drives → 2026-27 Activity Creation and click Open.

Saving it as a workspace

File → Save Workspace As...
Save the .code-workspace file somewhere easy (Desktop or Documents). Double-click it next time to reopen everything exactly as you left it.

Switching workspaces

File → Open Recent
Your recent folders and workspaces are listed here. Click one to jump to it.

Why it matters

Each time you switch workspaces, Claude's "world" changes. You can have one workspace per curriculum and keep context cleanly separated.

Two Ways to Talk to Claude

You'll use both. They're not competing — they do different jobs.

Slash commands (skills)

Pre-built workflows. Type / to see the list. Example: /move-to-review

Use for: pipeline actions — submitting, editing, finalizing. These run multi-step processes and update Jira for you.

Freeform requests

Just type what you want in plain English. Example: "Rewrite Q3 so it targets the climax instead of the rising action."

Use for: edits, questions, explanations. Treat Claude like a colleague you're talking to.

Skills are a subset of requests, not a replacement. Most of your work will be freeform conversation; you'll only run one skill (/move-to-review) per draft.

Anatomy of an Activity File

Every *_Activity_Content.md file has the same four sections. Knowing where to look saves time.

## ACTIVITY-LEVEL INFORMATION

Metadata + alignment + summary + reader content (the numbered text students see). You'll rarely edit this — and don't touch the numbered paragraphs, the question criteria point at them.

## CONTENT BLOCKS: INTRODUCTION

The Mojo-character intro bubbles students see at the start. Usually one or two short messages. Edit the wording freely if it feels off for the lesson.

## QUESTIONS

Where ~90% of your editing will happen. Each question has a type, prompt, answer/criteria, and (for highlight questions) paragraph ranges. This is the heart of the activity.

## CONTENT BLOCKS: TRANSITIONAL

Short Mojo-character messages that sit between questions to scaffold the flow — a quick recap, a nudge into the next concept, or a moment of encouragement. Edit the wording freely.

## CONTENT BLOCKS: TYPE 3

The closing/conclusion block. A short wrap-up after the final question. Edit freely.

Editing a Markdown File

Markdown (.md) is just plain text with a few formatting symbols. You can edit it like any document.

  1. Click in the editor and type. You can edit anything — change words, rewrite questions, delete lines. No special syntax needed for small edits.
  2. Save with Cmd + S. A white dot on the file tab means unsaved changes. After save, the dot disappears.
  3. Or ask Claude to edit for you. "Change Q3 from a TDQ to an LCQ" or "Tighten the criteria on Q2." Claude will make the edit directly in the file — you'll see it happen.
  4. Review Claude's edits before saving. Claude saves as it goes. If something's wrong, just tell it — "actually, revert that" or "make it shorter."
You don't need to memorize markdown syntax. Headings (##), bold (**), and lists (-) are the only things you'll see often, and Claude handles them.

Ask Claude vs. Edit by Hand

Both are valid. Picking the right one saves time. Rule of thumb: big or repetitive → Claude; small & surgical → just type it.

Ask Claude when...

  • You're rewriting a question or changing its type
  • You need to apply the same fix to multiple questions
  • You want to rework the criteria to match the match report
  • You need to check or adjust paragraph references
  • You're not sure how to phrase something — describe what you want
  • You want a sanity check ("does this match the learning goal?")

Just type it when...

  • You're fixing a typo or changing one word
  • You already know the exact wording you want
  • You're deleting something small
  • You're adjusting formatting (bold, bullet, heading)
If you're manually rewriting most of the draft, something's off with the inputs (module plan or source materials) — not the output. Flag it to Michaela so the skill can be improved.

Correcting Claude Mid-Task

You don't have to wait for Claude to finish before redirecting it. The feedback loop is always open.

Interrupt when it's wrong

Press Esc to stop Claude mid-response. Then redirect: "wait, that's not what I meant — let's try again with X."

Be specific

"Make Q3 shorter" is vague. "Cut Q3's prompt to one sentence and keep the criteria the same" is actionable. The more specific you are, the less guessing Claude does.

Show, don't just describe

Pasting an example or pointing at a line is faster than explaining. "Make Q3 more like Q1" works because Claude can read Q1 directly.

Ask "why" any time

"Why did you pick a TDQ here?" is a valid question. Claude will explain its reasoning — and sometimes you'll realize the reasoning was off.

Output #1: The Match Report

Claude's plan for the activity — what it analyzed, what it decided to target, and why. Read this before diving into the draft.

What you'll find

  • Lesson Requirements — activity type, grade, learning goal, verbatim inputs from your plan
  • Pedagogical Intent Analysis — analytical inventory, emphasis hierarchy (primary vs. supporting), key text sections
  • Excerpt Selection — which paragraphs and why
  • Time Budget — question count & estimated minutes vs. the lesson section replaced

What to look for

  • Assessment target — does it match the module plan's learning task?
  • Emphasis hierarchy — is Claude pointing at the right concept as PRIMARY?
  • Paragraph ranges — are the key text sections the ones you'd pick?
  • Time budget — does the question count fit the replaced section?
Match reports vary in length and format — some are lean, some dense with regeneration notes. Focus on what's there, not whether the shape matches every other report.

Output #2: The Activity Content

The actual draft Claude built. This is what you edit and what the student will see.

Where to focus your eye

  • Questions section — prompts, criteria, and question types
  • Criteria lines — do they point at the right paragraphs? Are they calibrated (not too loose, not too strict)?
  • Question sequence — does the cognitive demand build up sensibly?
  • Content block intro — does it orient students to the concept without giving away answers?

Common issues to watch for

  • Questions that describe what happens but don't name the concept
  • Criteria that are too vague ("correct response mentions...")
  • Paragraph references that don't actually contain the evidence
  • Question counts that drift off the time budget
  • Verbatim content from the plan not appearing in the draft

Strategic Reading: Use Both Together

The gold is in the comparison. Either document alone won't catch drift — but reading them side-by-side will.

Match report = the plan

What Claude intended to build. The strategy, the targets, the reasoning.

Activity content = the execution

What Claude actually built. The questions, criteria, and text students see.

Questions the comparison answers

  • Does the activity hit the primary emphasis the match report identified?
  • Are the question types the ones the report said would work — or did the draft drift?
  • Do the criteria point at the same paragraphs the report listed as key text sections?
  • Does the question count match the time budget the report planned for?
  • If the report says "PRIMARY: 4 highlight questions," does the draft actually have 4 highlights — or did it swap in free-response?
If the plan and the execution disagree, your edits should usually bring the execution in line with the plan — not the other way around.

Submitting With /move-to-review

When your edits are done, run this one skill. It's a short conversation, not a button press.

  1. Type /move-to-review in Claude Code.
  2. Confirm which draft — Claude lists the activities in draft_outputs/ and asks which one you're submitting.
  3. Answer "Any notes for the reviewer?" — this is your chance to flag uncertainty, ask a specific question, or give context the diff can't show. If you have nothing to add, just say so.
  4. Pick a reviewer — Meg Quick or Eric Snider. Claude will assign them in Jira.
  5. Claude handles the rest — it generates a change summary, moves files to review_ready/, transitions the Jira ticket to "Needs Lead Review," and @mentions your reviewer in the ticket comment so they get a notification.
The "notes for the reviewer" field is load-bearing. A one-sentence note like "I'm unsure if Q3 is too hard for grade 6 — thoughts?" gets way better review feedback than an empty submission.

Today vs. the Future

Today: The Google Doc gets heavy use — you're all looking at a concentrated batch of drafts for the first time. Expect to add a lot.

Future state: The Doc becomes an ongoing issues log you add to when you notice patterns. Breakout rooms go away — Slack DM or a standing check-in replaces them.

Best Practices & Pitfalls

Do

  • Trust but verify — skim the match report first, then the draft
  • Be specific when correcting Claude
  • Start a fresh conversation per lesson
  • Save often (Cmd + S) — or let Claude save for you
  • Flag recurring issues to Michaela so skills can improve
  • Use "why did you..." to understand Claude's reasoning

Don't

  • Manually rewrite everything — if you are, the inputs were wrong
  • Touch _DO_NOT_EDIT/ folders
  • Edit the numbered reader content paragraphs
  • Run one mega-conversation across many drafts
  • Let Claude run to a bad end — interrupt with Esc
  • Submit with an empty "notes for reviewer" if you have real questions
