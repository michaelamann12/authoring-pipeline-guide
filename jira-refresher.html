Quick Overview

Tickets are pre-created for you on your curriculum's Jira board. Your job is to find the lessons you want to work on, drag them through the early lanes, and let the pipeline handle the build. One Jira ticket = one lesson, named after the Activity ID.

Key Rules

A few things worth keeping in mind as you work:

  • One ticket = one lesson. Each ticket is named after the Activity ID (e.g. OE_SLP_XX_G9_1.1_7_v1).
  • Tickets are ready for you. Head to your curriculum's board to find them — no need to create your own.
  • Work in whatever batch size feels right. Pull a full module or just one ticket — both work great. Assigning to yourself makes the board easier to filter later.
  • Two things make builds happy: a matching Activity ID and source materials sitting in the Curriculum Hub. A quick check before you drag saves the most common failures.
  • Once a build is in motion, give it space. When a ticket is in Draft Activity with Claude, the pipeline will move it for you — no need to nudge it.
  • The module plan is your source of truth. Verbatim content flows through untouched, so the plan is the place to lock down anything that needs to read a specific way.

Step-by-Step Pipeline Walkthrough

1Find your tickets your curriculum's board

Go to your curriculum's Jira board. Tickets for your lessons are already waiting in Backlog. Filter by assignee or Activity ID prefix to locate the ones you want.

2Backlog → Authoring optional — you drag

If you'd like a staging lane for prep work — finalizing the module plan, confirming source materials in the Curriculum Hub, or just queueing up what you plan to build next — drag tickets into Authoring and assign them to yourself. If you're ready to build right away, you can skip this and go straight to step 3.

3Backlog or Authoring → Draft Activity with Claude you drag — this kicks off the build

Dragging a ticket into Draft Activity with Claude triggers the automated build. Claude runs six skills in order:

  1. /populate-lesson — pulls source materials into the lesson folder
  2. /extract-requirements — extracts lesson requirements and pedagogical intent
  3. /find-exemplars — searches the Activity Catalog for matching patterns
  4. /pick-excerpt — selects and extracts the text excerpt with correct numbering
  5. /generate-activity — generates the full activity content
  6. /edit-draft — runs a 17-point validation pass and silently fixes issues

The original draft is snapshotted into a _DO_NOT_EDIT/ folder before any edits, so you can always compare back to what Claude first produced.

Builds run in the background in a staggered queue — each lesson takes roughly 45–60 minutes of build time, but you don't wait on it. Drop your tickets and most will be drafted by the next morning. Do not re-drag, re-assign, or move tickets while they're working.

4Draft Activity with Claude → Draft Review automatic

When the build finishes, the pipeline moves the ticket to Draft Review on its own. The finished folder lands in draft_outputs/[Curriculum]/[DATE]_[Lesson_ID]/.

5Draft Review your core editing work

Open the draft in VS Code. You have three ways to edit:

  • Direct edits in the file
  • Ask Claude in plain English (e.g. "Question 3 feels too easy — push it harder")
  • Re-run /edit-draft after your edits if you want another 17-check validation pass

Check: curriculum alignment, scaffolding logic, paragraph references, verbatim preservation, grade-appropriate tone. The original draft is preserved in _DO_NOT_EDIT/ — you can't break anything.

6Draft Review → Needs Lead Review you run /move-to-review

When you're satisfied, type /move-to-review in the Claude panel. It:

  • Generates a change summary diffing your edits against the original
  • Asks for any notes you want to leave the reviewer
  • Saves a REVIEWED snapshot
  • Reconciles the Match Report and runs a final structural/numbering QA
  • Moves all files to review_ready/[DATE]_[Lesson_ID]/
  • Transitions the Jira ticket to Needs Lead Review and @-mentions the lead

Do not drag the ticket manually — the command handles it.

7Lead handles the rest

  • Lead approves → runs /move-to-final → generates a two-layer changelog (original → your edits → lead's edits), captures feedback, moves the file to final_outputs/[Curriculum]/[Lesson_ID]/, and transitions the ticket to In Transformation with an @-mention to the transformation team.
  • Lead requests changes → runs /revisions → transitions the ticket back to Draft Review, reassigns it to you, and posts the revision notes (with a Priority flag if urgent). Edit, then run /move-to-review again.

How Do I Know a Lesson Is Done?

Build step is complete when the ticket lands in the "Draft Review" lane. That's your signal that:

  • All six pipeline skills finished
  • The folder exists in draft_outputs/[Curriculum]/[DATE]_[Lesson_ID]/
  • The ticket is yours to edit

Builds run in the background in a staggered queue — each lesson takes roughly 45–60 minutes of build time, but you don't wait on it. Drop your tickets and most will be drafted by the next morning, so check back the next morning rather than watching one ticket. If a ticket is still in Draft Activity with Claude well past expected time (and earlier tickets in the same batch have already moved), the build likely failed — click the ticket and check the comments for any details about why it failed, or ping Michaela to take a look!

Watch the board for these transitions:

  • Backlog or Authoring → Draft Activity with Claude: you just triggered a build
  • Draft Activity with Claude → Draft Review: build succeeded, draft is ready
  • Draft Review → Needs Lead Review: /move-to-review ran cleanly
  • Needs Lead Review → Draft Review: a lead sent it back for revisions
  • Needs Lead Review → In Transformation: lead finalized via /move-to-final

Common Mistakes & Reminders

  • Creating new tickets. You don't need to — they're already on the board. If you can't find a ticket you expect, ping ops rather than making one.
  • Dragging a ticket out of Draft Activity with Claude manually. The pipeline moves it. If you drag it, you'll desync the ticket from the actual build state.
  • Resist the urge to re-drag. Builds run in the background in a staggered queue — each lesson takes roughly 45–60 minutes of build time, and most are drafted by the next morning. Re-triggering creates duplicate work and confuses the pipeline.
  • Activity ID mismatch. The Jira ticket name must match the module plan Activity ID exactly. One typo = no build.
  • Missing source materials. If the text isn't in the Curriculum Hub, the build fails. Verify before dragging.
  • Moving Draft Review → Needs Lead Review by dragging. Use /move-to-review — it generates the change summary, snapshots, QA pass, and lead @-mention. Dragging skips all of it.
  • Editing files in _DO_NOT_EDIT/. That's the original-draft backup for the audit trail. Edit the working file, not the snapshot.
  • Trust the lane. If the ticket is in Draft Review, your draft is ready. If it's still in build, wait.
