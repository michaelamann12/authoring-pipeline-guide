What is a Module Plan?

The module plan is a spreadsheet where authors define the blueprint for every activity in a module. It tells Claude what to build — the objective, text excerpt, question types, target task, and driving questions for each lesson.

The module plan is the single most important input to draft quality. A clear, well-structured plan produces a strong first draft. A vague plan produces a draft that needs heavy editing.

What Goes in the Plan

ColumnWhat to IncludeHow Claude Uses It
Activity ID The unique identifier (e.g., OE_SLP_XX_G9_1.1_7_v1) Names all output files and links to the Jira ticket. No ID = no draft.
Activity Objective What students will learn or do Anchors the entire activity — every question ladders up to this
Text Excerpt Reference to the source text and which section to use. For partial-page excerpts, specify boundaries: e.g., Chapter 2, pp. 15–18 or Chapter 2, from p.15 "opening words" to p.18 "closing words" Claude retrieves the text from the Curriculum Hub and extracts the specified section
Target Task (Verbatim) The assessment question exactly as you want it used — whether from the curriculum or your own wording Used word-for-word as the final assessment; Claude does not modify this. Use this OR Target Task Outline — not both.
Driving Questions (Verbatim) Scaffolding questions you want used exactly as written — from the curriculum or your own Used word-for-word in the question sequence
Driving Questions (Outline) Additional guidance — question topics, skills to target, sequencing hints Claude uses these as direction to generate additional scaffolding questions
TT Format/Type The format of the target task (e.g., Shortwrite HS, Longwrite HS) Determines the structure of the assessment section
Planning Notes Anything you want Claude to know when building this activity — context, guidance, constraints, connections Claude reads these notes as direct input. The more specific, the better the draft.

Planning Notes — Your Direct Line to Claude

Planning notes are a freeform field where you tell Claude anything relevant to this build. The more specific you are, the better the draft. Examples of what authors include:

  • Curriculum alignment: "Hit those TDQs closely" or "Use your judgement — adapt freely for better scaffolding"
  • Excerpt guidance: "The excerpt should emphasize the dialogue in chapter 5" or "Skip the first two pages — start where the conflict begins"
  • Connections: "This lesson needs to connect back to the module 2 assessment" or "Students will have already read chapters 1–4 in class"
  • Scope adjustments: "Skip the vocabulary focus" or "We need more scaffolding than usual — this is a challenging text for this grade level"
  • Specific content to include: Paste questions, tasks, or prompts you want Claude to use directly — anything pasted here will be treated as author intent

What to paste vs. what not to paste: DO paste specific questions, tasks, and prompts you want used in the activity. DON'T paste entire teacher edition pages or background context — Claude can find that in the source materials itself.

Verbatim vs. Outline — When to Use Each

Verbatim columns contain exact wording — from the curriculum or your own — that Claude must use word-for-word. Outline columns are your creative direction — guidance that Claude interprets to generate questions, transitions, and instructional framing.

For each row, use one or the other for a given element: if you have exact wording for the target task, put it in Target Task (Verbatim). If you want Claude to generate the target task from your guidance, put direction in the Outline column instead. You can mix across elements — e.g., verbatim target task + outline driving questions.

Novel Curricula — Excerpt Numbering

For novel curricula, the /pick-excerpt skill extracts your specified pages and applies paragraph numbering based on specified rules (these rules are still being finalized). This numbered excerpt is what students see and what questions reference. You don't need to number paragraphs yourself; just specify the page range in the Text Excerpt column.

The Assembly Line Metaphor

Think of the pipeline as a factory. The module plan is the blueprint, and the source materials are the raw parts — steel, windows, steering wheels. If the parts aren't on the line, the factory can't build the car. Claude will either fail gracefully or produce an incomplete draft — it will not make things up.

Automated Material Gathering

The /populate-lesson skill handles material discovery automatically:

  • Reads the module planning sheet to identify which text, which chapters, which lessons
  • Locates source texts in the Curriculum Hub (searches across 135+ titles)
  • Copies only the chapters referenced in the plan — not the whole book
  • Pulls teacher lesson plans, unit plans, and supplemental materials
  • Exports Google Sheets planning docs as CSV

Each curriculum organizes materials differently (Amplify uses letter-coded units, WW uses module numbers, EL has its own structure). The system knows how to navigate all of them.

The Curriculum Hub

All source texts live in the Curriculum Hub on Google Drive, organized by curriculum, grade, and module. This is where Claude looks when it needs to retrieve a text excerpt.

  • Anthology curricula (Amplify, HMH, MYP, StudySync): Steven's procurement process places texts into the correct folders. If procurement is complete for a module, the texts should be ready.
  • Novel/supplemental curricula (Odell, EL, Guidebooks): Authors may need to source materials themselves (e.g., a TED talk transcript, a public domain speech). These must be placed in the Hub before a build can run.

Before submitting a module plan for build: Verify that every text referenced in your plan actually exists in the Curriculum Hub. If it doesn't, either place it there yourself or flag it to Steven. A missing source text = an incomplete draft.

What Needs to Be in Place

  • The full text (or relevant excerpt) in the correct Hub folder
  • Any supplemental materials referenced in the plan (student handouts, tool pages, etc.)
  • File names that follow the naming convention: {Title}_{Excerpt or Full Text}.txt (e.g., Farewell to Manzanar_Full Text.txt or Dark They Were and Golden-Eyed_Excerpt.txt)

Common Questions

  • Materials used across multiple lessons: No need to copy files into each lesson folder. Source materials stay in their canonical Hub location and Claude references them from there. If a novel is already sourced from another curriculum, Claude will copy it over to the relevant text folders for the lesson it's working on.
  • Rubrics and writing task resources: Same — keep them in the Hub once. Claude finds them when referenced in the plan.
  • Multiple file formats (PDF, TXT, etc.): Claude reads whatever is available, preferring text files when both exist. You don't need to worry about duplicates.
  • Using only part of a text: Don't edit source files directly. Instead, specify the exact pages or section in your module plan's Text Excerpt column (e.g., Chapter 2, pp. 15–18). The /pick-excerpt skill handles extraction.
  • Anthology texts: The full text stays in the Hub even if you're only using part of it. Specify which section in your plan.

When Sources Are Missing

If Claude can't find a referenced text, the build will flag it rather than hallucinate content. The draft will come back incomplete, and the Jira ticket will reflect that source materials need attention before the draft can be completed.

Jira Structure

Work is tracked in the APT (Authoring Pipeline Testing) Jira project. The hierarchy:

  • Epic = one module (e.g., Odell G9 Module 1)
  • Story/Task = one lesson/activity (ticket name = Activity ID)

The Activity ID

Every activity has a unique ID (e.g., OE_SLP_XX_G9_1.1_7_v1) that follows it through the entire pipeline. This ID:

  • Names the Jira ticket
  • Names the output folder and all generated files
  • Links the module plan row to the draft output
  • Will connect to the Procurement Dashboard to show which activities are available for ticket creation

No Activity ID = no draft. The ID is how every system in the pipeline identifies and tracks an activity. It must be present in the module plan before a build can be triggered.

How Tickets Get Created

Tickets are created when procurement is complete for a module. Ops (Steven) confirms that source materials are ready, and tickets are created in bulk for the module. Each ticket's name is the Activity ID.

Your Jira Workflow (Authors & Leads)

Here's what you do with tickets day-to-day:

  1. Pull a full module/unit from Backlog into Ready for Authoring at once (not one ticket at a time)
  2. Assign tickets to yourself — this makes them easy to find on the board
  3. Drag to Drafting when you're ready for Claude to build — this triggers the pipeline
  4. Review when the ticket lands in Draft Review (Claude auto-advances it)
  5. Submit for lead review via /move-to-review when your edits are done
  6. Lead reviews and either sends back for revisions (via /revisions — reassigns ticket to author with feedback) or finalizes (via /move-to-final)

Draft Generation — What Claude Does
The automated steps that turn a plan into a draft
Automated Author Lead Product
▶

How It Works

When a Jira ticket moves to Draft Activity with Claude, Claude executes a sequence of specialized skills. Each skill handles one stage of the build, running in order:

StepSkillWhat It Does
1 /populate-lesson Reads the module plan, navigates the Curriculum Hub folder structure, copies source materials to the working folder. Handles 7+ curricula with different naming conventions.
2 /pick-excerpt Selects and extracts the exact text excerpt. Handles paragraph numbering rules: preserved numbers for anthology curricula (AMP, HMH, MYP, STS), sequential numbering for novel curricula (WNW, EL, FTK).
3 /draft-activity Generates the full activity: introduction, driving questions with criteria and correct-threshold judgments, transition text, assessment section (claim + write), and extension question.

What Claude Reads

  • The module plan row for this specific activity
  • The source text from the Curriculum Hub
  • Teacher lesson plan PDF (if available)
  • Curriculum-specific guidance documents (question type glossary, formatting rules)
  • 2–3 quality-matched exemplar activities from the 944+ exemplar catalog

Claude reads the same inputs your authors would read — the lesson plan, the text, the teacher materials. It synthesizes patterns from the exemplar catalog to build questions that match proven formats. The planning notes authors write directly shape the output.

What Claude Produces

A complete Activity_Content.md file containing:

  • Activity-level metadata (ID, objective, alignment, text summary)
  • Reader content (the formatted text excerpt with correct numbering)
  • Introduction blocks (Mojo chat bubbles)
  • Driving questions — each with question type, criteria for teachers, correct threshold, and 3 example correct responses (JBCs)
  • Transition text between sections
  • Assessment section (claim question + written response)
  • Extension question
  • Conclusion blocks

Plus a Match_Report.md documenting which exemplars were used, why certain questions were chosen, and how the activity maps to the standard.

What Claude will never do: Modify verbatim content from the module plan (target tasks, verbatim driving questions), fabricate text excerpts from sources it can't find, or override the plan's specifications for question counts and content. The module plan is the source of truth.

What Determines Draft Quality

Strong inputs produce strong drafts

  • Clear objective → questions that build toward a coherent learning arc
  • Specific excerpt reference → well-scoped text that supports the questions
  • Detailed outline column → scaffolding that matches the author's pedagogical intent
  • Precise planning notes → the right level of creative latitude

Weak inputs produce weak drafts

  • Vague objective → unfocused question sequence
  • Missing excerpt info → wrong or overly broad text selection
  • Empty outline column → generic scaffolding that may not match intent
  • Missing source text → incomplete or failed draft

A Second Pair of Eyes

After the draft is generated, the /edit-draft skill runs an 11-point validation pass in a completely separate context from the generation step. This is like having a second reviewer who hasn't seen the work before — they catch things the first pass might have missed.

The 11-Point Checklist

  1. Verbatim fidelity — questions match module plan word-for-word where specified
  2. Paragraph references — every cited paragraph actually exists in the text
  3. Question progression — proper scaffolding sequence (accessible → analytical → assessment)
  4. Assessment alignment — Claim + Write match curriculum scope
  5. Excerpt completeness — no placeholders, full text present
  6. Line number accuracy — for poetry and drama
  7. Criteria quality — text-specific, not generic
  8. Brevity limits — question stems, criteria, and content blocks stay concise and within word count guidelines
  9. Format compliance — no excluded fields, required sections present
  10. MC formatting — proper option formatting for multiple choice
  11. Content blocks — Introduction, Transition, Conclusion all present

Issues are fixed silently. The author receives a polished draft with these checks already resolved. You can also re-run /edit-draft manually at any time during your review if you want a fresh validation pass.

The Author's Role

Claude produces a first draft — not a finished product. The author's job is to bring curriculum expertise, pedagogical judgment, and knowledge of students to refine that draft into something that works in a real classroom.

Author focuses on...

  • Pedagogical judgment calls
  • Question nuance & text sensitivity
  • Adjusting difficulty calibration
  • Adding curriculum-specific context
  • Final approval

System already handled...

  • File discovery & organization
  • v22 formatting compliance
  • Paragraph reference accuracy
  • Question scaffolding structure
  • Match report documentation

Finding Your Draft

Drafts live in the draft_outputs/ folder, organized by date:

What to Look For

AreaCheck
Alignment Does the activity match the objective and the curriculum's intent for this lesson?
Question Sequence Does the question arc build toward the right thinking? Do driving questions scaffold logically toward the target task?
Question Types Are the question types appropriate? (highlight, TDQ, LCQ, etc.)
Text References Do paragraph references match the actual excerpt? Are questions grounded in the text, not generic?
Criteria & Thresholds Are the correct-threshold criteria fair and the example responses realistic for this grade level?
Verbatim Fidelity Are target tasks and verbatim driving questions unchanged from the plan?
Assessment Alignment Does the assessment align with the end-of-module task?
Tone & Language Is the student-facing language grade-appropriate and engaging?

Your expertise is the value add. Claude handles structure, formatting, and initial question generation. You bring the deep understanding of the text, the students, and what actually works in a classroom. That combination is what makes the pipeline powerful.

Tips for Effective Editing

  • Start by reading the draft top to bottom without editing — get a feel for the arc
  • Check the Match Report if a design choice surprises you — it explains the reasoning
  • Be specific with Claude — "Make this better" is hard to act on; "The criteria should reference the metaphor in paragraph 8" gives Claude exactly what it needs
  • The draft is a starting point. Make it yours — rewrite questions, change types, reorder, add, or remove as needed
  • You can't break anything. The _DO_NOT_EDIT/ folder has a backup of the original

Review Path

After editing, submit your draft for lead review:

  1. Author edits the draft (direct edits, Claude assistance, or /edit-draft)
  2. Author submits via /move-to-review — generates a change summary and routes to your lead
  3. Lead reviews in review_ready/ and either:
    • Sends back for revisions via /revisions — reassigns the Jira ticket to the author with specific feedback and an @mention notification (supports a priority flag for urgent items)
    • Finalizes via /move-to-final — captures feedback, generates changelog, moves to transformation

Adding Content Blocks

If you need to add content blocks (Introduction, Transition, Conclusion, etc.) that aren't in the draft, you can either add them directly in the markdown following the existing format, or ask Claude: "Add a transition block before Question 4 that bridges the vocabulary discussion to the analytical focus."

Running /move-to-final

After lead review is complete, the lead types /move-to-final in the Claude panel. This skill:

  1. Asks you for feedback — What worked well? What needed the most editing? Any patterns you're seeing across drafts?
  2. Generates a changelog — Compares your edited version against the original draft to document every change (two-layer diff if lead review was used)
  3. Moves the final version to final_outputs/ so the transformation team can pick it up
  4. Updates the Jira ticket status to In Transformation
  5. Cleans up the draft folder (removes backup files, archives the Match Report)

Your feedback is captured in the changelog and helps improve future drafts. Even a one-sentence note like "Questions were solid but the intro felt generic" is valuable. Patterns in feedback directly inform how Claude drafts for this curriculum going forward.

Why This Step Matters

Simply dragging a Jira ticket loses the "why." The finalization step captures:

  • A record of what changed and why
  • Patterns that help improve future drafts (e.g., "Claude consistently under-scaffolds for this curriculum")
  • Accountability and transparency in the review process
  • Qualitative data that informs skill improvements upstream

From Draft to Deliverable

Once a ticket moves to In Transformation, the finalized activity content is converted into the format required for the platform. The /format-csv skill produces a transformation-ready CSV and a formatted Google Doc for Reader Content.

Current Process

  1. A finalized Activity_Content.md is tagged for transformation via the Jira ticket
  2. The content is converted into a structured CSV or JSON ready for review
  3. The transformation team (Jennifer & Sabrina) reviews the output for completeness and formatting
  4. Once approved, the ticket moves to Done

What the Transformation Team Receives

  • A consistently structured file with all required fields populated
  • Activity metadata, reader content, questions, criteria, and content blocks in a predictable format
  • The Activity ID linking everything back to the Jira ticket and module plan

Consistency is the goal. The upstream pipeline produces output in a standard format so that transformation doesn't have to handle one-off variations. When the format is predictable, QA is faster and errors are fewer.
