Ticket Lifecycle

How a Ticket Moves Across the Board

Each transition is triggered by either a human action, a Claude Code skill, or the automated polling pipeline. Jira's simplified workflow allows any status → any status (global transitions).

TransitionTriggered ByTransition IDWhat Happens
Backlog → Authoring Author (manual drag) 71 Author picks up the ticket for manual authoring work
Backlog → Draft Activity with Claude Author or Product (manual drag) 111 Ticket queued for automated drafting. Must have 26-27-pipeline label.
Draft Activity with Claude → Draft Review process-tickets.sh (automated) 121 Pipeline completes draft, moves ticket, removes pipeline_queued label
Draft Review → Needs Lead Review /move-to-review skill 141 Assigns to lead reviewer, @mentions them in comment
Needs Lead Review → Draft Review /revisions skill 121 Lead sends back for revisions, @mentions original author, reassigns to them
Needs Lead Review → Final Review Lead (manual drag) 131 Lead approves the draft for final QA pass
Final Review → In Transformation /move-to-final skill 41 Activity finalized, @mentions Jennifer Chien and Sabrina Rourke Frost
In Transformation → Done Transformation team (manual) 31 CSV/JSON delivered and verified

Labels & Tags

Jira Labels Used by the Pipeline

LabelPurposeAdded ByRemoved By
26-27-pipeline Required for the polling pipeline to pick up a ticket. Without this label, a ticket in "Draft Activity with Claude" status will be ignored. Author or Product (manual) Never removed
pipeline_queued Marks a ticket as currently being processed by the pipeline. Prevents duplicate processing on the next polling cycle. process-tickets.sh process-tickets.sh (on success or failure)
source_text_missing Pipeline ran but couldn't find the source text for this lesson. Draft is incomplete. Requires manual intervention to add source materials. process-tickets.sh Manual (after source text added)
priority_revision Lead flagged this revision as urgent — needs author attention today. /revisions skill Manual (after revisions complete)
25-26 Legacy label on 25-26 tickets. Pipeline ignores these (only looks for 26-27-pipeline). Historical N/A

Claude Code skills are the primary interface for moving tickets through the authoring pipeline. Each skill handles both the file operations AND the Jira state changes, so authors never need to manually drag tickets or tag people.

Author Discovery

The skill reads the ticket's ?expand=changelog and finds the most recent transition into status 10671 (Needs Lead Review). The author field of that changelog entry is the person who submitted the draft — that's who gets tagged and reassigned.

Automated Polling Pipeline

How process-tickets.sh Works

The pipeline runs locally on a Mac via a LaunchAgent. It polls Jira for tickets that are ready for automated drafting.

Polling Query (JQL)

project = ET26
AND status = "Ready for Authoring"
AND labels = "26-27-pipeline"
AND labels NOT IN ("pipeline_queued")

The status name in Jira is "Ready for Authoring" (ID 10636) — this maps to the board column "Draft Activity with Claude." The query requires the 26-27-pipeline label and excludes tickets already being processed.

Pipeline Sequence Per Ticket

Add pipeline_queued label → /populate-lesson → /pick-excerpt → /draft-activity → /edit-draft

Outcome Handling

OutcomeJira ActionLabel Changes
Success Transition → Draft Review (ID 121) Remove pipeline_queued
Source text missing Stays in Draft Activity with Claude + Jira comment explaining what's missing Remove pipeline_queued, Add source_text_missing
Failure Stays in Draft Activity with Claude + Jira comment with failure summary (generated by Claude) Remove pipeline_queued

Notification Model

How People Get Notified

All notifications use Jira's native @mention system via ADF (Atlassian Document Format) mention nodes in comments. Plain text "@Name" does NOT trigger notifications. When @mentioned, users receive:

ChannelDefaultNote
Jira bell notificationAlways onIn-app, visible on all Jira pages
Email notificationOn by defaultCan be disabled per-user in Jira notification settings

Who Gets Notified When

EventWhoVia
Draft ready for reviewSelected lead (Meg or Eric)/move-to-review → @mention + assign
Revisions requestedOriginal author/contractor/revisions → @mention + reassign
Ready for transformationJennifer Chien + Sabrina Rourke Frost/move-to-final → @mention both
Pipeline failureVisible on ticketprocess-tickets.sh → Jira comment (no @mention) + macOS notification to pipeline operator

Authentication

Jira API Credentials

All Jira-integrated skills require a ~/.jira-config file on the user's machine. This uses each person's own Jira API token so comments and assignments are attributed to the right person.

# ~/.jira-config
JIRA_DOMAIN=coursemojo.atlassian.net
JIRA_EMAIL=your.email@coursemojo.com
JIRA_API_TOKEN=your_token_here

Setup Steps

  1. Go to Atlassian API Token Manager
  2. Click "Create API token" — name it anything (e.g., "Claude Code")
  3. Create ~/.jira-config with the three lines above
  4. Verify: source ~/.jira-config && curl -s -u "$JIRA_EMAIL:$JIRA_API_TOKEN" "https://$JIRA_DOMAIN/rest/api/3/myself"

Cost: Free. API tokens are included with every Jira account.
If missing: Skills will warn the user and skip Jira updates. File operations still complete normally.

@mention Comment Format (ADF)

Plain text "@Name" does NOT trigger Jira notifications. Comments must use ADF mention nodes:

{
  "body": {
    "type": "doc", "version": 1,
    "content": [{
      "type": "paragraph",
      "content": [
        {
          "type": "mention",
          "attrs": {
            "id": "ACCOUNT_ID",
            "text": "@Display Name",
            "accessLevel": ""
          }
        },
        {"type": "text", "text": " — your message here."}
      ]
    }]
  }
}
