Set up your machine

Two one-time setup steps and you're ready to author. Your team will have installed VS Code and Claude Code for you — these steps connect them to your work.

Setup 1

Claude Code in VS Code

Claude Code lives inside VS Code (the program you'll edit in). Open it by clicking the Claude icon in the left sidebar. Three things to know on day one:

📁

The folder you open = Claude's whole world

Claude can only see files inside the folder you've opened. Open your curriculum's folder with Cmd + O (Mac) before you start.

💬

One conversation per lesson

Start a fresh chat for each activity so Claude doesn't mix up lessons. Past chats stay available if you need them.

⎋️

Esc stops Claude

Press Esc any time to interrupt and redirect — you never have to wait for it to finish.

Setup 2

Connect your Jira account (API key)

This lets Claude move your tickets and notify reviewers on your behalf when you run /move-to-review. You'll do it once.

  1. Create your Jira API token. Go to id.atlassian.com/manage-profile/security/api-tokens → click Create API token → name it Claude Code → Create → Copy. Keep that browser tab open — you can't see the token again after you close it.
  2. Save it to a config folder. Mac: On your Desktop, make a folder named config. Open TextEdit, click Format → Make Plain Text (important!), paste the two lines below, and save it inside config as jira-token.txt.
    Windows: Make a config folder on your Desktop, right-click inside → New → Text Document → name it jira-token.txt, open in Notepad, paste the two lines, save.     
    email: your.name@coursemojo.com
token: <paste the token you copied in step 1>
  3. Let Claude finish the setup. Open Claude Code, paste the prompt below into the chat, and press Enter. Claude reads your token file, configures everything, and tests the connection — you'll see a success message in a few seconds. 
    Please set up my Jira credentials for the move-to-review skill.

Read the file at ~/Desktop/config/jira-token.txt — it contains two lines:
  email: <my email>
  token: <my Jira API token>

Parse those values, then create (or overwrite) the file at ~/.jira-config with
exactly this format, substituting my email and token:

JIRA_DOMAIN=coursemojo.atlassian.net
JIRA_EMAIL=<my email>
JIRA_API_TOKEN=<my token>

Then verify the credentials work by running:
  curl -s -o /dev/null -w "%{http_code}" -u "<email>:<token>" https://coursemojo.atlassian.net/rest/api/3/myself

If it returns 200, reply "Jira setup complete — you can now use /move-to-review."
If it returns anything else, tell me the response code and what might be wrong
so I can fix the token file.
Your token is a password. Don't share your config folder or the file inside it.
The big picture

How an activity gets made

Every activity travels the same path. Some stages are automated; some are yours. Knowing which is which is most of the job.

✏️

1. You plan it

You write the module plan — the blueprint that says what each activity should teach, which text to use, and how many questions of which type. This is the single biggest driver of draft quality.

⚙️

2. Claude drafts it

When you drag the ticket to Draft Activity with Claude, the pipeline reads your plan, gathers the source text, picks the excerpt, and writes a complete first draft — then runs an automatic quality pass.

🔎

3. You refine it

You review the draft in Claude Code, tighten questions and criteria, then send it to your lead. Your lead approves it, and it moves on to the transformation team.

Two things must be right or the build fails

1. The module plan

Claude reads it as its blueprint. The Activity ID is how it knows what to build — no Activity ID, nothing happens.

2. The source materials

The text and teacher materials must be in the Curriculum Hub. If Claude can't find them, the build flags it rather than inventing content — the draft comes back incomplete.

What happens "behind the scenes" when Claude drafts (optional — you never run these yourself)

When a ticket hits Draft Activity with Claude, the pipeline runs six steps in order — reading your plan, pulling out the requirements, finding similar high-quality examples, selecting the exact excerpt, writing the activity, and running a quality check. You don't trigger or manage any of this; it's listed only so the process isn't a black box.

The one move that matters most: dragging a ticket into Draft Activity with Claude kicks off the build. Claude takes over and, about 45 minutes to an hour later (longer if you start several at once), the ticket lands in Draft Review on its own. Give it space — once a build is in motion, the pipeline moves the ticket for you.

✅ Do

  • Pull tickets from Backlog/To-Do and assign them to yourself
  • Confirm your module plan and source materials are ready before you drag to build
  • Drag into Draft Activity with Claude to start a build
  • Pull one ticket or a whole module — batch size is up to you
  • Use /move-to-review to send a finished draft to your lead

⛔ Don't

  • Create new tickets — they already exist on your board
  • Manually drag a ticket out of Draft Activity with Claude — it desyncs the build
  • Re-drag a ticket to "rebuild" — builds are staggered and take time
  • Manually drag from Draft Review to Needs Lead Review — let /move-to-review do it
  • Let the ticket name and the module plan's Activity ID disagree
Your rhythm

Your daily loop

Once you're set up, every activity follows the same six steps.

  1. Find your tickets.Open your curriculum's Jira board and look in Backlog / To-Do. Filter by your name or the Activity ID prefix.
  2. Pull & assign.Drag the ticket(s) to To-Do or Authoring and assign them to yourself. Use Authoring for any prep — finalizing the module plan, confirming source materials.
  3. Trigger the build.Drag the ticket to Draft Activity with Claude. Claude builds the draft (~45 min–1 hr) and moves it to Draft Review automatically.
  4. Review & edit in Claude Code.Open the draft folder, read the activity against the plan, and refine. Edit directly, ask Claude in plain English, or run /edit-draft for an automated quality pass.
  5. Send it to your lead.Run /move-to-review. Claude asks which draft, asks for any notes for the reviewer, lets you pick Meg or Eric, then moves the ticket to Needs Lead Review and notifies them.
  6. Watch for the outcome.If your lead requests changes, you'll get a Jira notification with their notes and the ticket returns to Draft Review. If approved, your lead finalizes it — you're done with that activity.
The "notes for reviewer" field is worth using. Even one sentence — "I'm unsure if Q3 is too hard for grade 6" — makes the review faster and better.
In Claude Code

Reading & editing a draft

When a draft lands, its folder holds a few files. You only edit one of them.

*_Activity_Content.md
Edit this

The actual activity students will see. This is where ~90% of your editing happens.

*_Match_Report.md
Read this

Claude's plan and reasoning — the target analysis, what to emphasize, and why. Read it to understand the draft's intent.

Excerpt selection notes
Reference

Why Claude chose the paragraphs it did. Handy if a paragraph reference looks off.

_DO_NOT_EDIT/
Leave alone

Backups the pipeline uses to track your changes. Don't touch this folder.

Edit by hand when…

  • Fixing a typo or a single word
  • You already know the exact wording
  • Deleting a small piece or fixing formatting

Click into the file, type, and save with Cmd + S. A white dot on the tab means unsaved changes.

Ask Claude when…

  • Rewriting a question or changing its type
  • Applying the same fix across several questions
  • Reworking criteria or adjusting paragraph references

Be specific: "Cut Q3's prompt to one sentence and keep the criteria" beats "make it shorter." Review Claude's changes before moving on.

Read the Match Report and the activity together. The Match Report is the plan; the activity is the execution. Your edits should pull the execution toward the plan — check that the draft hits the main emphasis, the question types match, the criteria point at real evidence, and the question count fits the time budget.
When something's off

If a build fails or you're stuck

⚠️

A build failed

The ticket stays in Draft Activity with Claude and a Jira comment explains why — almost always missing source materials or an Activity ID mismatch. Fix the cause, then re-drag.

📝

Draft issue vs. tool pattern

Just this draft? Fix it in Claude Code. Noticing Claude get the same thing wrong across lessons? That's a tool pattern — flag it so the skill improves for everyone.

💬

Not sure?

Ask. A quick question to Michaela or your lead beats guessing — especially in your first few activities.
